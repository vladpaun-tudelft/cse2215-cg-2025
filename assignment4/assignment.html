Ray

A ray is defined by its origin, a direction and a parameter t indicating the intersection point. In the code it is defined as follows:

struct Ray {
    glm::vec3 origin { 0.0f };
    glm::vec3 direction { 0.0f, 0.0f, -1.0f };
    float t { std::numeric_limits<float>::max() };
};

The intersection point of the ray is then defined as p = ray.origin + ray.direction * ray.t.

For every intersection method, the ray is passed by reference. You should check for intersections and update the ray.t value when necessary. Note that ray.t should always store the closest intersection parameter found so far, taking into consideration that rays should not intersect points behind the origin (only positive values for t).

In short:

  1. no intersection is found or \(t\leq 0\): return false
  2. found intersection point p
    1. if p is closer to the ray origin then current intersection point: return true and update ray.t
    2. else return false

Intersection tests for collision detection

In this first part of the assignment, we want to demonstrate the potential of intersection tests for scene analysis.

For this purpose, we consider a collision detection scenario that is often encountered in scene analysis for autonomous vehicles, traffic scenarios, or the interaction between robots. In this task, however, we consider collision detection for analyzing whether Pac-Man will be caught by 2 ghosts for a given motion in a particularly specified scenario depicted in the figure below.

For this task, we assume that both Pac-Man and the ghosts have a constant velocity of 1 tile/s in both x and y direction: When changing the direction of motion, the orientations of the shapes of Pac-Man and the ghosts remain unaffected. Under this constraints, the main objective we have to solve is to check whether following the path marked in yellow will allow Pac-Man to escape without colliding with any of the ghosts on their paths (marked in red).

Although you might infer it, the 2D depiction of the scenario itself does not directly provide insights on the solution, since the temporal behavior is not illustrated. Therefore, we now lift the representation to a 3D scenario where the 3rd dimension corresponds to time. This allows us to consider the motion of a particular scene point in the form of rays that pass straight through the 3D space and are redirected when the underlying object changes its direction of motion. While drawing this scenario gives us direct information on whether we get a collision with the ghosts, we can additionally use intersection tests to compute where a collision may occur.

4. Computation and visualization of motion behavior

This part will be implemented in pacman.cpp

Compute the motion for all the vertices of Pac-Man and the ghosts for a time intervall of [0s, 8s]. For a linear motion segment, i.e. a segment where Pac-Man or the ghosts respectively do not change their direction, you can compute the displacement of the vertices given at the beginning of the time interval according to:

\[ \begin{pmatrix} x \cr y \cr z \cr \end{pmatrix} = \begin{pmatrix} x_0 \cr y_0 \cr z_0 \cr \end{pmatrix} + t * \begin{pmatrix} speed_x \cr speed_y \cr -1 \cr \end{pmatrix} \]

5. Intersection tests for collision detection

The following part is to be implemented in ray_tracing.cpp.

Based on the spatio-temporal mesh structures we have obtained from the previous steps for Pac-Man and the ghosts, we will now conduct a collision test. For this purpose, you will consider the rays defined based on the sampled shape of Pac-Man and intersect them with the triangles defining the hull geometry of the ghosts’ motion.

5.1 Compute triangle plane

Start by computing the plane defined by three vertices of a triangle.

Plane trianglePlane(const glm::vec3& v0, const glm::vec3& v1, const glm::vec3& v2)

The trianglePlanereceives the three vertices of a triangle and defines the plane that contains the triangle, that is, you need to set the plane.normal and plane.D values. The Plane structure is defined as follows:

struct Plane {
    float D = 0.0f;
    glm::vec3 normal { 0.0f, 1.0f, 0.0f };
};

where D and normal are defined as in the ray tracing lecture.

5.2 Compute the intersection of a ray with a plane

Once you have the plane you should test for ray-plane intersection:

bool intersectRayWithPlane(const Plane& plane, Ray& ray)

This method intersectRayWithPlane receives the plane and the ray, returns false if the ray does not intersect the plane, or true otherwise. If an intersection is found, it should also update the ray.t value. In case of an intersection, the next step is to verify if the intersection point is inside the triangle.

5.3 Test whether a point is within a certain triangle

bool pointInTriangle(const glm::vec3& v0, const glm::vec3& v1, const glm::vec3& v2, const glm::vec3& n, const glm::vec3& p)

This method receives the three vertices of the triangle, the triangle normal, and the point to be verified p. It returns true if the point is inside the triangle, false otherwise. Note that a point on the triangle edge is considered as inside the triangle.

5.4 Ray-triangle intersection

Finally, you are ready to join the three previous function:

bool intersectRayWithTriangle(const glm::vec3& v0, const glm::vec3& v1, const glm::vec3& v2, Ray& ray)

The method intersectRayWithTriangle receives the three vertices of a triangle and the ray, and should check the intersection in three steps. First, compute the plane containing the triangle. Second, compute the intersection point of the ray and the plane. If there is an intersection, the third step is to check if the point is inside the triangle.

Tip Don’t forget to roll back the ray.t value modified in the intersectRayWithPlane method if the point is not inside the triangle.

Obs The methods will be checked separately by the unit tests, so do not implement everything inside the intersectRayWithTriangle method!

Your triangle intersection tests should cover the general case. If you want to debug/inspect your intersection tests with examples other than the hull geometry or the unit test cases, you can implement a test scene and corresponding draw and intersect methods and activate the test scene in the UI.
